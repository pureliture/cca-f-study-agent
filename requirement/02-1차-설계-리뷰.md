아래 평가는 GitHub의 **en 버전 CCA-F 자료**와 첨부하신 기존 설계안을 함께 기준으로 했습니다. 저장소 기준으로는 `guide_en.MD`와 `pdf/guide_en.pdf`가 있고, README에서도 “Practical Exercises”를 통해 tool design, MCP integration, structured output, context management, reliability를 연습하라고 안내합니다. en 가이드는 Claude Code, Claude Agent SDK, Claude API, MCP를 중심 기술로 보고, 시험 도메인도 Agent architecture, Tool/MCP, Claude Code, Prompt/Structured output, Context/Reliability로 구성되어 있습니다.

## 결론

첨부 설계의 방향은 좋습니다. 특히 **“이론(PDF) ↔ 실습(Lab) ↔ 복습(노트)”를 1:1로 연결**하고, NotebookLM에는 PDF와 인덱스를 올리며, 코드는 Jupyter/VS Code에서 관리한다는 분리는 타당합니다.

다만 CCA-F에 그대로 적용하기에는 한 가지 큰 문제가 있습니다. 기존 설계의 예시 모듈이 `intro-to-prompting`, `running-the-eval`, `model-based-grading` 중심인데, CCA-F en 가이드는 **Claude API, tool_use, Agent SDK, MCP, Claude Code, CI/CD, structured output, context/reliability, scenario architecture**가 중심입니다. 그래서 폴더 구조 철학은 유지하되, **모듈 축을 CCA-F 도메인/챕터/시나리오 기준으로 재설계**하는 것이 좋습니다.

제 평가는 다음과 같습니다.

| 항목            | 평가                                                   |
| ------------- | ---------------------------------------------------- |
| 학습 흐름 설계      | 좋음. PDF, Lab, Note 분리는 유지할 만함                        |
| NotebookLM 활용 | 좋음. 단, “코드 원문”보다 “코드 위치/의도/실습 인덱스”를 올리는 방식이 적합       |
| CCA-F 적합성     | 보완 필요. 현재 예시 모듈명이 CCA-F en 자료 구조와 맞지 않음              |
| 코드 관리성        | 보완 필요. 노트북만으로는 재현성, 테스트, 버전 관리가 약함                   |
| 업데이트 대응       | 방향은 좋지만 upstream commit, PDF hash, KR/EN 매핑이 추가되어야 함 |

## 1. 기존 설계에서 유지할 부분

기존 설계의 핵심인 다음 네 가지는 그대로 유지하는 것이 좋습니다.

첫째, **PDF는 NotebookLM용 읽기 전용 소스**로 두고, 코드는 Git/Jupyter/VS Code에서 편집하는 분리입니다. NotebookLM은 로컬 코드를 실행하거나 최신 상태를 자동 반영하는 도구가 아니므로, 코드 자체를 지식 소스로 삼기보다 “이 개념의 실습은 어느 경로에 있다”는 인덱스를 제공하는 방식이 맞습니다.

둘째, **번호 동기화**는 매우 유용합니다. 다만 CCA-F에서는 단순 `01`, `02`보다 `C01`, `D01`, `S01`처럼 의미 있는 prefix를 쓰는 편이 더 좋습니다. CCA-F는 챕터, 시험 도메인, 시나리오가 서로 겹치기 때문입니다.

셋째, **`notebooklm-source-index.md`를 함께 업로드**하는 방식은 좋습니다. 첨부안에서도 이 파일을 통해 NotebookLM이 “PDF 내용과 실습 폴더의 관계”를 알 수 있게 설계되어 있는데, 이 접근은 CCA-F에도 그대로 유효합니다.

넷째, **개인 노트 분리**도 좋습니다. PDF에 직접 메모를 섞으면 NotebookLM 재업로드, 원본성 관리, 번역본/영문본 비교가 어려워집니다.

## 2. 기존 설계의 핵심 문제

가장 큰 문제는 **모듈 설계가 CCA-F en 자료의 실제 구조와 다르다**는 점입니다.

첨부안의 예시는 다음 흐름입니다.

```text
01-intro-to-prompting
02-running-the-eval
03-model-based-grading
04-code-based-grading
```

이 구조는 “프롬프트/eval 강의”에는 잘 맞지만, CCA-F에는 부족합니다. CCA-F en 자료는 다음 주제를 핵심으로 둡니다.

```text
Claude API
tool_use / JSON schema / structured output
Claude Agent SDK
subagents / hooks / sessions
MCP tools / resources / servers
Claude Code / CLAUDE.md / skills / planning mode
CI/CD
context management / reliability
exam scenarios
```

따라서 기존 폴더 구조를 그대로 가져가면, NotebookLM에서 “MCP Resources 실습은 어디 있지?”, “Claude Code의 CLAUDE.md hierarchy 실습은 어디 있지?”, “Customer Support Agent 시나리오는 어느 실습과 연결되지?” 같은 질문에 답하기 어려워집니다.

또 하나의 문제는 **노트북 중심 설계가 CCA-F의 모든 실습에 맞지는 않는다**는 점입니다. Claude API나 structured output은 Jupyter 노트북이 잘 맞지만, Claude Code, `.claude/skills`, `.mcp.json`, GitHub Actions, CI/CD 실습은 노트북보다 **프로젝트 디렉터리, 샘플 앱, 설정 파일, 테스트 코드**가 더 적합합니다.

## 6. 실습 코드는 “노트북 + src + tests” 조합으로 관리

첨부 설계는 실습을 주로 `.ipynb`로 보고 있는데, CCA-F에서는 이 방식만으로는 부족합니다. 추천은 다음 구조입니다.

```text
03-labs/C02-tool-use-structured-output/
├── README.md
├── notebooks/
│   └── lab-01-tool-schema-design.ipynb
├── src/
│   ├── tools.py
│   ├── schemas.py
│   └── validators.py
├── tests/
│   ├── test_schemas.py
│   └── test_validators.py
├── data/
│   ├── extraction_cases.jsonl
│   └── expected_outputs.json
└── solutions/
    ├── tools_solution.py
    └── lab-01-tool-schema-design-solution.ipynb
```

이렇게 하면 역할이 분명해집니다.

| 파일/폴더        | 역할                         |
| ------------ | -------------------------- |
| `notebooks/` | 개념 설명과 단계별 실행              |
| `src/`       | 재사용 가능한 실습 코드              |
| `tests/`     | 이해 여부 검증                   |
| `data/`      | 입력/출력 샘플                   |
| `solutions/` | 정답 또는 참고 구현                |
| `README.md`  | PDF/NotebookLM/시험 도메인과의 연결 |

기존 설계의 `sys.path.append('../_shared')` 방식은 간단하지만 장기적으로는 깨지기 쉽습니다. 대신 `04-shared/cca_f_labs`를 패키지처럼 만들고 editable install 방식으로 쓰는 편이 좋습니다.

```bash
pip install -e .
```

또는 `uv`를 쓴다면:

```bash
uv pip install -e .
```

## 10. 최종 권장안

현재 설계는 **방향성은 8/10**입니다. 하지만 CCA-F에 직접 적용하는 설계로 보면 **6.5/10** 정도입니다. 이유는 구조 철학은 좋지만, 모듈명이 CCA-F en 자료와 맞지 않고, 코드 재현성/테스트/환경 관리가 부족하기 때문입니다.

최종적으로는 이렇게 가져가면 좋습니다.

```text
NotebookLM = KR PDF + 인덱스 + 도메인/시나리오/용어 매핑
Git/VS Code = 실습 코드 + 노트북 + 테스트 + 샘플 프로젝트
Notes = 개인 요약 + 오답 + 시험 시나리오별 판단 기준
Archive = KR PDF/EN guide/upstream commit/hash 기준 버전 관리
```

즉, 기존 설계의 핵심 문장인 **“이론(PDF) ↔ 실습(Lab) ↔ 복습(노트)”**는 유지하되, CCA-F에서는 이를 다음처럼 확장하는 것이 가장 적합합니다.

```text
KR PDF in NotebookLM
   ↕
NotebookLM source index / lab catalog / domain map / scenario map
   ↕
CCA-F labs: API, tool_use, Agent SDK, MCP, Claude Code, CI/CD, reliability
   ↕
src + tests + notebooks + scenario capstones
   ↕
personal notes / exam review / glossary
```

이 방식이면 NotebookLM에서 이론을 질문하다가도 “이 개념은 어느 실습에서 구현했지?”, “이건 시험 도메인 몇 번이지?”, “Customer Support Agent 시나리오와 연결되는 코드는 어디지?”를 자연스럽게 추적할 수 있습니다.
