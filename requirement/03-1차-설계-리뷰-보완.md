결론부터 말하면, **현재 제안했던 CLAUDE.md는 부족합니다.**
“학습 repo 기본 운영 규칙”으로는 괜찮지만, 네가 말한 수준의 **학습 진척도 시각화 + 모의고사 제출 분석 + 합격점 도달률 대시보드 + Claude Code design plugin 기반 UI 설계**까지 하려면 CLAUDE.md를 한 단계 더 구체화해야 합니다.

기존 설계의 “PDF는 NotebookLM, 실습 코드는 repo, 연결 정보는 index” 구조는 그대로 유지하면 됩니다. 이 방향은 첨부 설계의 핵심인 **이론(PDF) ↔ 실습(Lab) ↔ 복습(노트)** 1:1 연결 구조와도 맞습니다. 

## 1. 현재 CLAUDE.md의 부족한 점

현재 버전은 대략 이런 수준이었습니다.

```text
- sources는 읽기 전용
- question-bank는 문제 관리
- labs는 실습 코드
- 모든 문제는 domain/scenario/answer/explanation 포함
- lab은 README/src/tests 포함
- API key 커밋 금지
```

이건 기본 규칙으로는 충분하지만, 대시보드를 만들려면 아래가 추가되어야 합니다.

| 필요한 항목               | 현재 CLAUDE.md 상태 | 보완 필요                                                   |
| -------------------- | --------------- | ------------------------------------------------------- |
| 학습 진척도 데이터 모델        | 없음              | `progress.json`, `lab_status.json`, `attempts.jsonl` 필요 |
| 점수 계산 기준             | 없음              | scaled score, pass mark, 도메인 가중치 명시                     |
| 모의고사 제출 형식           | 없음              | answer sheet/import format 필요                           |
| 대시보드 산출물             | 없음              | static dashboard or React dashboard 명시                  |
| 시각화 요구사항             | 없음              | 도메인별, 시나리오별, 합격점 대비, 취약 영역                              |
| Claude Code skill 역할 | 일부만 있음          | design/build/import/analyze skill 분리 필요                 |
| 문제 품질 검수 기준          | 기본만 있음          | “정답 하나”, “distractor 검증”, “unofficial 표시” 강화 필요         |

CCA-F en 자료 기준으로 시험 도메인은 5개이고, 통과 기준은 720/1000으로 정리되어 있습니다.  또 Claude Code skill 기반 모의고사 예시인 `cert-exam.md`도 scaled score를 `100 + 정답률 * 900` 방식으로 계산하고 pass mark를 720으로 둡니다.  따라서 네 대시보드도 일단 이 계산식을 기본값으로 두면 됩니다.

## 3. 대시보드에서 보여줘야 할 핵심 지표

대시보드는 복잡하게 시작할 필요 없습니다. 처음부터 중요한 지표만 잡으면 됩니다.

### 1) 합격점 도달률

```text
현재 최고 점수: 690 / 1000
합격 기준: 720 / 1000
부족 점수: 30점
합격점 도달률: 95.8%
```

계산식:

```text
scaled_score = round(100 + (correct / total) * 900)
pass_gap = 720 - scaled_score
pass_progress = scaled_score / 720
```

### 2) 도메인별 준비도

CCA-F 도메인 가중치를 반영해서 보여주는 게 좋습니다.

```text
D1 Agent Architecture         72%    weight 27%
D2 Tool Design & MCP          64%    weight 18%
D3 Claude Code Workflows      80%    weight 20%
D4 Structured Output          70%    weight 20%
D5 Context/Reliability        58%    weight 15%
```

단순 정답률만 보면 안 되고, **시험 가중치 기반 readiness**를 같이 보여줘야 합니다.

```text
weighted_readiness =
  D1_accuracy * 0.27 +
  D2_accuracy * 0.18 +
  D3_accuracy * 0.20 +
  D4_accuracy * 0.20 +
  D5_accuracy * 0.15
```

### 3) 시나리오별 취약점

CCA-F는 시나리오 기반 문제라서 도메인만 보면 부족합니다.

```text
S01 Customer Support Agent              약함
S02 Code Generation with Claude Code    보통
S03 Multi-Agent Research System         강함
S04 Developer Productivity Tools        보통
S05 Claude Code CI/CD                   약함
S06 Structured Data Extraction          강함
S07 Conversational AI Architecture      약함
```

### 4) 개념 태그별 오답률

가장 중요한 지표입니다.

```text
취약 개념 TOP 5
1. subagent explicit context passing
2. MCP structured error
3. tool_choice any vs auto
4. CLAUDE.md hierarchy
5. progressive summarization risk
```

### 5) 실습 진행률

```text
C01 Claude API                         완료
C02 Tool Use / Structured Output        진행 중
C03 Agent SDK Orchestration             미시작
C04 MCP Integration                     진행 중
C05 Claude Code Workflows               완료
C06 CI/CD Headless                      미시작
C07 Reliability                         미시작
```

## 6. Claude Code design plugin은 어디에 쓰면 좋은가

네 말대로 **대시보드는 Claude Code design plugin을 쓰기 좋습니다.** 다만 design plugin이 맡을 일과 engineering 작업을 분리하는 게 좋습니다.

```text
design plugin
= 대시보드 UX 설계
= 정보 구조
= 화면 와이어프레임
= 컴포넌트 구성
= 학습자가 봐야 할 핵심 지표 우선순위
= 상태/빈 화면/error state 설계

engineering / 일반 Claude Code
= 데이터 스키마
= score 계산 로직
= JSON 변환
= React 구현
= 테스트
= dashboard-data export
```

Claude Code에 줄 프롬프트는 이렇게 잡으면 됩니다.

```text
Claude Code design plugin을 사용해서 CCA-F 학습 대시보드 UX를 설계해줘.

목표:
- 사용자가 모의고사를 제출하면 현재 합격점 720 기준으로 얼마나 도달했는지 보여준다.
- 도메인별/시나리오별/개념 태그별 취약점을 보여준다.
- 실습 코드 진행률과 모의고사 성과를 함께 보여준다.
- “다음에 무엇을 공부해야 하는지”가 한눈에 보이게 한다.

산출물:
1. 06-dashboard/dashboard-spec.md
2. 06-dashboard/wireframe.md
3. 06-dashboard/component-map.md
4. 06-dashboard/data-contract.md

주의:
- 아직 구현하지 말고 design artifact만 작성해라.
- 데이터는 04-exam-runner/export_dashboard_data.py가 생성하는 dashboard-data.json을 읽는 전제로 설계해라.
- NotebookLM/PDF는 dashboard에서 직접 읽지 않는다.
```

## 7. CLAUDE.md 개선안

현재 CLAUDE.md는 아래처럼 확장하는 것을 추천합니다.

```markdown
# CCA-F Study Repository Instructions

This repository manages Claude Certified Architect — Foundations study materials, mock exams, hands-on labs, and learning progress dashboards.

## Core principle

Separate responsibilities:

- NotebookLM is for KR PDF and theory Q&A.
- This repository is for labs, mock exams, progress data, and dashboards.
- Claude Code manages repository artifacts, but must not claim generated materials are official exam content.

## Source policy

- `01-sources/kr/` stores the user's KR PDF copy.
- `01-sources/en/` stores the English guide and reference materials.
- `01-sources/external-mocks/` stores cloned unofficial GitHub mock exam resources.
- External mock questions are unofficial and must be marked with source metadata.
- Do not import or generate leaked real exam content.
- Do not claim any generated or community question is official.

## Question bank policy

All questions must conform to `04-exam-runner/question_schema.json`.

Required fields:
- `id`
- `source`
- `domain`
- `scenario`
- `difficulty`
- `stem`
- `choices.A`
- `choices.B`
- `choices.C`
- `choices.D`
- `answer`
- `explanation.correct`
- `explanation.why_others_wrong`
- `concept_tags`
- `status`

Allowed statuses:
- `imported_unreviewed`
- `generated_unreviewed`
- `reviewed_ok`
- `needs_fix`
- `rejected`

Rules:
- Each question must have exactly one best answer.
- Distractors must be plausible but clearly wrong.
- Explanations must explain why the correct answer is right and why each distractor is wrong.
- Scenario-based questions are preferred over abstract recall questions.
- Duplicates must be flagged instead of silently merged.

## Exam attempt policy

All submitted mock exam attempts must conform to `04-exam-runner/attempt_schema.json`.

Each attempt must record:
- attempt ID
- source
- submitted timestamp
- pass mark
- selected answers
- correct answers
- domain
- scenario
- concept tags
- correctness

Default scoring:
- `scaled_score = round(100 + (correct / total) * 900)`
- `pass_mark = 720`

Dashboard calculations must include:
- raw score
- scaled score
- pass/fail
- gap to pass mark
- domain accuracy
- weighted readiness
- scenario accuracy
- weak concept tags
- recent trend

## Domain weights

Use these CCA-F domain weights:

- D1 Agentic Architecture & Orchestration: 27%
- D2 Tool Design & MCP Integration: 18%
- D3 Claude Code Configuration & Workflows: 20%
- D4 Prompt Engineering & Structured Output: 20%
- D5 Context Management & Reliability: 15%

## Lab policy

Every lab under `03-labs/` must include:

- `README.md`
- `src/`
- `tests/`
- related domain mapping
- related scenario mapping when applicable
- no hardcoded secrets
- mock clients by default
- clear validation command

Labs should connect theory to implementation.

## Learning progress policy

Learning progress is tracked in `05-learning-data/`.

Required files:
- `study-progress.json`
- `lab-status.json`
- `wrong-answer-log.md`
- `attempts/*.json`

Claude Code must update these files when:
- a lab is completed
- a mock exam attempt is submitted
- weak areas are analyzed
- dashboard data is exported

## Dashboard policy

Dashboard artifacts live under `06-dashboard/`.

The dashboard must answer:

1. Am I currently above or below the pass mark?
2. How far am I from 720?
3. Which domains are weak?
4. Which scenarios are weak?
5. Which concept tags are repeatedly missed?
6. Which labs should I do next?
7. Is my score trend improving?

Dashboard implementation rules:
- Use `06-dashboard/data/dashboard-data.json` as the source of truth.
- Do not read PDFs directly.
- Do not mutate question bank data from the dashboard.
- Keep dashboard generation reproducible from local JSON files.
- Prefer static dashboard first; add server only if needed later.

## Claude Code skill usage

Use design-oriented skills for:
- dashboard UX
- information architecture
- wireframes
- component map
- visual hierarchy

Use implementation-oriented skills for:
- schema creation
- scoring logic
- JSON import/export
- tests
- dashboard app implementation

## Safety

- Do not commit API keys.
- Do not call paid APIs unless explicitly requested.
- Prefer fixtures and mocks for labs.
- Do not overwrite reviewed questions without explicit instruction.
- Do not modify KR PDF source files.
```

## 9. 최종 판단

현재 CLAUDE.md는 **기본 repo 운영용으로는 60점**, 네가 말한 목표까지 포함하면 **부족합니다**.

네 목표는 단순히 “공부 자료 정리”가 아니라 아래에 가깝습니다.

```text
CCA-F 학습 운영 시스템
= 이론 자료 관리
+ 문제은행 관리
+ 모의고사 제출/채점
+ 오답 분석
+ 실습 코드 진행률
+ 합격 가능성 대시보드
+ 다음 학습 추천
```

그래서 CLAUDE.md도 단순한 코딩 규칙이 아니라, **학습 데이터 운영 규약 + 대시보드 데이터 계약 + Claude Code 작업 경계**까지 포함해야 합니다.

추천 순서는 이겁니다.

```text
1. CLAUDE.md 확장
2. question_schema.json 작성
3. attempt_schema.json 작성
4. submit_attempt.py / score_attempt.py 작성
5. dashboard-data.json export 작성
6. design plugin으로 dashboard spec/wireframe 작성
7. 정적 dashboard 구현
8. React/Vite dashboard로 확장
```

처음부터 멋진 UI를 만들기보다, 먼저 **모의고사 제출 → 점수 계산 → 합격점 gap → 도메인별 취약점 → dashboard-data.json** 흐름을 고정하는 게 핵심입니다. UI는 그다음에 design plugin으로 다듬는 게 맞습니다.
