CCA-F Study Runtime 유스케이스 지도

요구사항 기준의 사용 시나리오는 “자료를 읽고 끝내는 학습”이 아니라, 문제 생성/검수 → 모의고사 풀이 반복 → 점수/취약점 분석 → 대시보드 → 다음 학습 행동으로 이어지는 로컬 학습 운영 루프다. PDF와 NotebookLM은 근거/이론 계층이고, 실제 런타임 판단은 JSON/JSONL 파일에서만 일어난다.

1. 한 줄 요약: 사용자는 무엇을 반복하나

01

이론 출처 고정

guide_en.pdf는 읽기 전용 근거로 등록한다. 런타임은 PDF를 파싱하지 않는다.

02

문제 세트 생성/검수

새 문제는 generated/draft로 만들고, 검증/리뷰 후 seed 또는 reviewed pool로 승격한다.

03

모의고사 풀이 제출

사용자가 선택지만 담은 answers JSON을 만들고, 이를 normalized attempt로 변환한다.

04

점수 계산

정답률에서 scaled score를 계산하고 720 기준 pass/fail/gap을 산출한다.

05

대시보드 생성

attempt들과 lab status를 모아 dashboard-data.json 하나로 만든다.

06

다음 학습 결정

취약 도메인/시나리오/개념 태그를 보고 다음 실습과 복습을 정한다.

2. 반복 모의고사 운영 루프

사용자는 모의고사를 한 번 제출하고 끝내지 않는다. 학습이 진행될수록 새로운 문제 세트를 만들고, 이미 푼 attempt는 누적 보존하며, 대시보드는 최신 점수와 장기 추세를 함께 보여줘야 한다.

문제 후보 생성

PDF/NotebookLM/학습 노트를 근거로 새 문제 후보를 만든다. MVP에서는 자동 생성이 아니라 수동/외부 생성 결과를 repo에 들여오는 흐름이다.

generated draft

검증/리뷰

schema, source id, scenario id, duplicate id, official 금지 규칙을 통과해야 출제 pool로 들어간다.

validate_questions status != official

모의고사 세트 선택

도메인 균형형, 취약 개념 집중형, 전체 랜덤형처럼 목적별 question set을 고른다.

D1-D5 mix weak tags

반복 풀이 저장

각 풀이 결과는 append-only attempt로 남긴다. 같은 문제라도 날짜/세트/라벨이 다른 시도는 별도 기록이다.

attempt-001 attempt-002

누적 추세 갱신

dashboard-data는 모든 attempts를 다시 집계해 최신 점수, 반복 오답, 추세, 다음 lab을 갱신한다.

trend weak concepts
분석 결과가 다음 문제 세트의 출제 비중과 복습 계획으로 되돌아간다.
중요한 경계: MVP는 PDF 자동 파싱/자동 문제 생성을 구현하지 않는다. 하지만 운영 모델은 반복 생성을 전제로 둔다. 즉, 새 문제는 02-question-bank/generated/ 또는 별도 reviewed pool로 들어오고, 검증 전에는 절대 official로 취급하지 않는다.

5. 데이터가 이동하는 방식

근거 계층

source-register.md, domain-map.md, scenario-map.md는 문제와 분석의 기준점이다.

문제 계층

seed, generated, reviewed 문제 pool이 있다. 출제 전에는 schema와 source/scenario 검증을 통과해야 한다.

풀이 계층

정규화된 attempt는 각 답에 domain/scenario/concept_tags를 포함한다. 이후 scoring은 문제은행을 다시 읽지 않는다.

↓

분석 계층

raw score, scaled score, pass gap, domain/scenario/concept breakdown을 계산한다.

표시 계층

dashboard-data.json이 정적 dashboard의 유일한 입력이다. HTML은 데이터를 직접 만들지 않는다.

학습 행동 계층

weak concept와 lab-status.json를 결합해 다음 실습 후보를 고른다.

6. 문제 세트와 attempt가 쌓이는 구조

Seed bank

초기 기준 문제. 최소 10개, D1-D5 커버리지 확보. 회귀 테스트와 샘플 모의고사의 안정 기준이다.

Generated bank

새로 만든 후보 문제. 자동/수동 출처와 관계없이 official 금지, 중복/시나리오/source 검증 대상이다.

Attempt history

attempt_label과 finished_at로 매번 별도 attempt를 만든다. 덮어쓰지 않고 누적한다.

Trend dashboard

최신 attempt만 보지 않고 모든 attempts의 scaled score, weak concepts, domain drift를 시간순으로 보여준다.

8. 유스케이스별 성공/실패 경로

UC-1: 새 문제 후보 생성

성공: 새 문제는 source/domain/scenario/concept_tags/status를 가진 JSONL row로 저장된다.

실패: 근거가 불명확하거나 자동 생성 결과를 official로 표시하면 출제 pool에 넣지 않는다.

UC-2: 문제은행 검증

성공: 모든 문제가 schema, source, scenario, duplicate rule을 통과한다.

실패: unknown source, unknown scenario, duplicate id, generated/official status가 발견되면 중단한다.

UC-3: 반복 모의고사 제출

성공: 매 회차 answers JSON이 question bank와 join되어 새로운 attempt JSON으로 저장된다.

실패: 알 수 없는 question_id는 실패하고, 누락 답안은 null/incorrect로 기록된다.

UC-4: 점수 계산

성공: round(100 + accuracy * 900)로 scaled score를 만들고 720 기준 gap을 계산한다.

실패: attempt schema가 깨졌거나 answers가 없으면 정의된 edge case로 처리한다.

UC-5: dashboard export

성공: 모든 attempt와 lab status를 모아 deterministic JSON을 만든다.

실패: 빈 attempts라도 top-level key는 유지하고 empty state를 만든다.

UC-6: dashboard 확인

성공: 정적 HTML이 ../data/dashboard-data.json만 fetch해서 7개 위젯을 렌더링한다.

실패: 데이터가 비어도 JS error 없이 no data 상태를 보여준다.

UC-7: 다음 문제 세트 조정

성공: weak concept와 domain breakdown을 보고 다음 모의고사 세트의 출제 비중을 조정한다.

실패: 최근 1회 점수만 보고 전체 학습 방향을 바꾸지 않는다. trend와 반복 오답을 같이 본다.

UC-8: KR PDF 확장

성공: 기존 EN-CORE를 유지하고 KR-LECTURES source set을 추가한다.

실패: NotebookLM upload package를 runtime source of truth로 착각하지 않는다.

9. 구현 Phase와 유저 가치의 연결

Phase 1

뼈대/메타

출처와 도메인/시나리오 기준이 생긴다.

Phase 2

패키지

CLI 모듈을 로컬에서 실행할 수 있다.

Phase 3

스키마

문제/attempt 계약이 고정된다.

Phase 4

문제 검증

seed bank 품질을 자동 확인한다.

Phase 5

제출 정규화

사용자의 답안이 분석 가능한 attempt가 된다.

Phase 6

채점

합격 여부와 취약점이 계산된다.

Phase 7

Export

학습 상태가 dashboard-data로 집계된다.

Phase 8

Dashboard

다음 학습 행동을 한 화면에서 결정한다.

현재 사용자가 직접 체감하는 기능은 Phase 5부터 시작된다. Phase 1-4는 데이터 품질과 계약을 만드는 기반이고, Phase 5-8이 “이번 attempt가 이전보다 나아졌는가, 다음 문제 세트는 무엇이어야 하는가”라는 반복 학습 경험을 완성한다.
