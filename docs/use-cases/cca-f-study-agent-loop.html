CCA-F 학습 루프 에이전트 구성도

이 문서는 유스케이스가 아니라, CCA-F 학습 시스템을 지속적으로 완성해가기 위한 운영 아키텍처 지도다. 여기서 “에이전트”는 반드시 독립 실행 AI 프로세스를 뜻하지 않는다. MVP에서는 대부분 CLI와 파일 계약으로 구현되고, 이후 자동화가 붙을 때 각 역할이 실제 agent/skill/script로 분리될 수 있다.

1. 전체 루프: 관찰 → 진단 → 결정 → 실행 → 기록

Source Indexer

출처와 시험 구조를 고정한다

source-register, domain map, scenario map, NotebookLM index를 관리한다.

Question Curator

문제 후보를 검증 가능한 은행으로 만든다

문제 생성, 외부 문제 staging, 중복/출처/상태 검증을 담당한다.

Exam Composer

오늘 풀 문제 세트를 고른다

전체 균형형, 취약 개념 집중형, 회차 복습형 세트를 구성한다.

2. 에이전트 역할별 계약

Source Indexer

입력: source PDF, source register, domain/scenario maps

출력: 안정적인 source id와 시험 구조

결정: 어떤 출처가 active인지, 어떤 source id를 문제에 허용할지

Question Curator

입력: seed/generated/reviewed JSONL

출력: 검증 통과한 출제 가능 문제 pool

결정: draft, unofficial, reviewed, rejected 상태

Attempt Normalizer

입력: question bank, learner answers

출력: domain/scenario/tag가 포함된 attempt JSON

결정: unknown question 실패, missing answer는 incorrect 처리

Scoring Analyst

입력: normalized attempt

출력: raw/scaled score, pass gap, breakdowns

결정: 720 기준 합격 여부와 약점 ranking

Progress Aggregator

입력: all attempts, lab-status

출력: deterministic dashboard-data.json

결정: latest attempt, trend, recommended_next

Study Planner

입력: weak concepts, lab progress, trend

출력: 다음 mock exam 방향과 lab/복습 후보

결정: 반복 오답을 줄이는 다음 행동

External Intake Agent

입력: community repo/site, user-provided files

출력: provenance가 있는 generated JSONL

경계: post-MVP. 자동 scraping은 MVP 밖이다.

Question Generator

입력: 학습 노트, weak concepts, 사람이 지정한 범위

출력: draft/unofficial question candidates

경계: 생성 결과는 검증/리뷰 전 출제 확정 아님

Dashboard Designer

입력: dashboard-data contract, learner questions

출력: wireframe, component map, static dashboard

경계: MVP에서는 vanilla HTML/CSS/JS만 사용

5. MVP와 이후 자동화의 경계

MVP에서 구현하는 루프

local JSON/JSONL 기반으로 validate → submit → score → export → dashboard를 완성한다.

no network no PDF parsing static dashboard append-only attempts

post-MVP로 미루는 자동화

외부 문제 intake, 문제 자동 생성, duplicate similarity, source-site parser, React dashboard는 MVP 완료 후에 붙인다.

external intake question generation crawler/parser dashboard v2

항상 지켜야 하는 안전선

generated/community 문제를 official로 표시하지 않는다. 출처/라이선스/검수 상태 없이 practice pool에 넣지 않는다.

status != official provenance required review gate

업데이트할 때 보는 기준

새 기능은 어느 에이전트 역할에 속하는지, 입력/출력 artifact가 무엇인지, dashboard의 어떤 learner question을 개선하는지로 판단한다.

role contract artifact feedback

6. 앞으로 구조를 완성할 때의 체크리스트

역할 추가

새 자동화가 생기면 먼저 어느 agent role인지 정한다. 기존 역할에 들어가면 새 역할을 만들지 않는다.

입출력 고정

각 역할은 파일/JSON 계약으로 연결한다. 암묵적으로 PDF나 웹페이지를 읽는 흐름은 만들지 않는다.

검증 게이트

생성/가져오기/정규화 단계마다 실패 조건을 먼저 둔다. 특히 source, status, duplicate를 확인한다.

피드백 연결

대시보드 결과가 다음 문제 세트, lab 추천, 복습 노트 중 어디로 돌아가는지 명확히 연결한다.
